SIADLP — Flujo de Distribución

Sistema Integral de Administración, Distribución y Logística de Papa — La Cosecha S.A.C.

1. Problemática

La Cosecha S.A.C. es una distribuidora de papas que abastece pollerías en Lima. Actualmente su proceso de distribución es completamente manual:

  • Los pedidos se registran en papel o por WhatsApp, sin trazabilidad ni historial
  • No hay un sistema para agrupar pedidos por ruta y asignar vehículos/choferes de forma eficiente
  • Las entregas no se registran en tiempo real — no se sabe qué se entregó, qué quedó pendiente, ni qué tuvo problemas
  • No existe visibilidad del estado operativo del día (cuántos pedidos hay, cuántos se despacharon, cuántos se entregaron)
Alcance del TAP: El sistema se enfoca exclusivamente en el proceso de distribución — desde que el cliente solicita productos hasta que se confirma la entrega en su local. No cubre abastecimiento (compras a proveedores), procesamiento (producción/pelado) ni almacenamiento (gestión de inventario).

3. Detalle de Cada Etapa

3.1 Registro de Pedido

El operador registra el pedido del cliente indicando qué productos necesita y en qué cantidad.

flowchart TD
    A[Operador abre módulo Pedidos] --> B[Selecciona cliente]
    B --> C["Agrega líneas:\nproducto + cantidad (kg)"]
    C --> D[Selecciona fecha de entrega]
    D --> E["Pedido creado en estado REGISTRADO"]
    E --> F["Log: REGISTRADO por usuario X"]

    style E fill:#1e3a5f,stroke:#60a5fa,color:#fff
Datos del pedido: Cliente, fecha de entrega, lista de productos con cantidades en kg, y observaciones opcionales. El sistema no maneja precios — la distribución se enfoca en qué llevar, cuánto y a dónde.

3.2 Confirmación del Pedido

flowchart TD
    A["Pedido en estado REGISTRADO"] --> B{Decisión del operador}
    B -->|Aprobar| C["Estado → CONFIRMADO"]
    B -->|Rechazar| D["Estado → CANCELADO\n(requiere motivo)"]
    C --> E["Pedido disponible para despacho"]
    D --> F["Pedido cerrado"]

    style C fill:#1a3a1a,stroke:#4ade80,color:#fff
    style D fill:#3a1a1a,stroke:#f87171,color:#fff
¿Por qué confirmar? No todos los pedidos registrados se despachan. El operador valida disponibilidad antes de confirmar. Los pedidos cancelados requieren un motivo para trazabilidad.

3.3 Preparación y Despacho

Los pedidos confirmados se organizan por ruta para crear hojas de carga eficientes.

flowchart TD
    A["Operador selecciona fecha de despacho"] --> B["Sistema agrupa pedidos\nCONFIRMADOS por ruta"]
    B --> C["Operador arma la hoja de carga:\n• Selecciona ruta\n• Asigna vehículo\n• Asigna chofer\n• Elige pedidos a incluir"]
    C --> D["Se crea Hoja de Carga\nestado: PREPARANDO"]
    D --> E["Operador confirma el despacho"]
    E --> F["Estado hoja → DESPACHADO\nPedidos → DISPATCHED\nSe crean entregas PENDIENTE"]
    F --> G["Chofer inicia la ruta"]
    G --> H["Estado hoja → EN_RUTA"]

    style D fill:#3a3a1a,stroke:#facc15,color:#fff
    style F fill:#1e3a5f,stroke:#60a5fa,color:#fff
    style H fill:#2a1a3a,stroke:#c084fc,color:#fff
Validaciones del despacho:
  • Un pedido solo puede pertenecer a una hoja de carga
  • Solo pedidos en estado CONFIRMADO pueden incluirse
  • El peso total (kg) no puede exceder la capacidad del vehículo
  • La ruta, el vehículo y el chofer deben estar activos en el sistema

3.4 Entrega y Cierre

El chofer registra el resultado de cada entrega en el sistema.

flowchart TD
    A["Hoja de carga EN_RUTA"] --> B["Por cada pedido en la hoja:"]
    B --> C{¿Se entregó exitosamente?}
    C -->|Sí| D["Entrega: ENTREGADO\nPedido: DELIVERED"]
    C -->|No| E["Entrega: NOVEDAD\nPedido: ISSUE\nRegistra observación del problema"]
    D --> F{¿Todas las entregas\ncompletadas?}
    E --> F
    F -->|Sí| G["Hoja de Carga → COMPLETADO\n(automático)"]
    F -->|No| H["Quedan entregas pendientes"]

    style D fill:#1a3a1a,stroke:#4ade80,color:#fff
    style E fill:#3a3a1a,stroke:#facc15,color:#fff
    style G fill:#1a3a1a,stroke:#4ade80,color:#fff
Cierre automático: Cuando TODAS las entregas de una hoja de carga han sido registradas (ya sea como ENTREGADO o NOVEDAD), el sistema transiciona la hoja automáticamente a estado COMPLETADO.

4. Máquina de Estados del Pedido

stateDiagram-v2
    [*] --> REGISTERED : Crear pedido
    REGISTERED --> CONFIRMED : Confirmar
    REGISTERED --> CANCELLED : Cancelar (motivo)
    CONFIRMED --> DISPATCHED : Despachar (hoja de carga)
    DISPATCHED --> DELIVERED : Entrega exitosa
    DISPATCHED --> ISSUE : Novedad / problema
    ISSUE --> CONFIRMED : Reprogramar
    DELIVERED --> [*]
    CANCELLED --> [*]
EstadoSignificadoQuién lo cambia
REGISTEREDPedido creado, pendiente de confirmaciónSistema (al crear)
CONFIRMEDAprobado, disponible para despachoOperador
DISPATCHEDIncluido en una hoja de carga, en caminoMódulo despacho
DELIVEREDEntregado exitosamente al clienteChofer
ISSUEProblema en la entrega (cliente ausente, rechazo, etc.)Chofer
CANCELLEDPedido cancelado antes del despachoOperador
Reprogramación: Un pedido con NOVEDAD puede volver a estado CONFIRMADO para ser incluido en un despacho futuro.

5. Máquina de Estados de la Hoja de Carga

stateDiagram-v2
    [*] --> PREPARANDO : Crear hoja de carga
    PREPARANDO --> DESPACHADO : Confirmar despacho
    DESPACHADO --> EN_RUTA : Chofer inicia ruta
    EN_RUTA --> COMPLETADO : Todas las entregas registradas
    COMPLETADO --> [*]
EstadoSignificado
PREPARANDOSe están seleccionando los pedidos y asignando recursos
DESPACHADOConfirmado, listo para salir
EN_RUTAEl chofer está realizando las entregas
COMPLETADOTodas las entregas fueron registradas
